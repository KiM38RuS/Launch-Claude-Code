## Разрешенные действия

### Файловые операции
- Создавай резервные копии при критических изменениях

### Тестирование изменений
- После внесения изменений в ClaudeCodeLauncher.ahk **всегда** запускай скрипт для проверки:
  ```bash
  rm -f ahk_error.log && start "" "ClaudeCodeLauncher.ahk" && sleep 5 && cat ahk_error.log 2>/dev/null || echo "Ошибок не обнаружено"
  ```
- Проверяй лог на наличие ошибок интерпретатора
- Убедись, что GUI открывается корректно
- Проверь основные функции (восстановление сессий, кнопки, диалоги)

#### Инструкция по созданию бэкапов

**Когда создавать бэкап:**
- Перед внесением значительных изменений в основной скрипт
- Перед рефакторингом или добавлением новых функций
- При достижении стабильной версии (перед началом новой фичи)

**Как создавать бэкап:**
1. Скопировать текущий `ClaudeCodeLauncher.ahk` в папку `Backup/`
2. Переименовать копию, добавив версию: `ClaudeCodeLauncher X.Y.Z.ahk`
3. Версия берётся из константы `SCRIPT_VERSION` в коде (например, "v1.2.0")
4. Формат имени: `ClaudeCodeLauncher 1.2.0.ahk` (без префикса "v")

**Команда для создания бэкапа:**
```bash
# Получить текущую версию из скрипта
version=$(grep 'SCRIPT_VERSION := "v' ClaudeCodeLauncher.ahk | sed 's/.*"v\(.*\)".*/\1/')
# Создать бэкап
cp ClaudeCodeLauncher.ahk "Backup/ClaudeCodeLauncher $version.ahk"
```

**Пример структуры Backup/:**
```
Backup/
├── ClaudeCodeLauncher 1.0.ahk      (первая версия)
├── ClaudeCodeLauncher 1.1.0.ahk    (добавлена новая функция)
├── ClaudeCodeLauncher 1.1.1.ahk    (исправлен баг)
└── ClaudeCodeLauncher 1.2.0.ahk    (текущая стабильная версия)
```

#### Инструкция по обновлению версии

**Система версионирования (X.Y.Z):**
- **X (major)** - мажорные изменения: проект становится полностью функционален или новый функционал доведён до рабочего состояния
- **Y (minor)** - минорные изменения: накопилось большое количество небольших изменений, проект в удовлетворительном виде
- **Z (patch)** - патчи: довольно небольшие изменения, исправления, улучшения
- **Суффикс 'a' (альфа)** - присваивается автоматически при создании версии, снимается после тестирования и исправления всех найденных ошибок (или если ошибок не обнаружено)

**Процесс обновления версии:**

1. **Создать бэкап текущей версии** (см. инструкцию выше)

2. **Определить новую версию:**
   - Новая версия всегда получает суффикс 'a': `1.3.0a`
   - После тестирования и исправления багов суффикс снимается: `1.3.0`

3. **Обновить версию в трёх местах:**
   - Строка 11: `;@Ahk2Exe-SetVersion X.Y.Za` (с суффиксом 'a')
   - Строка 19: `SCRIPT_VERSION := "vX.Y.Za"` (с суффиксом 'a')
   - Заголовок окна обновится автоматически

4. **Обновить CHANGELOG.md:**
   ```markdown
   ## vX.Y (YYYY-MM-DD)
   - Описание изменений (только видимые пользователю)
   - Не засоряй техническими деталями реализации
   - Фокусируйся на функциональности и пользовательском опыте
   ```

5. **После тестирования - снять суффикс 'a':**
   - Обновить версию в тех же трёх местах: `X.Y.Z` (без 'a')
   - Создать финальный бэкап стабильной версии

6. **Создать git commit:**
   ```bash
   git add ClaudeCodeLauncher.ahk CHANGELOG.md
   git commit -m "Bump version to X.Y.Za"
   # После тестирования:
   git commit -m "Release version X.Y.Z"
   ```

**Примеры:**
- `1.2.0 → 1.2.1a` - начало работы над исправлением бага
- `1.2.1a → 1.2.1` - баг исправлен и протестирован
- `1.2.1 → 1.3.0a` - начало работы над новой функцией
- `1.3.0a → 1.3.0` - функция готова и протестирована
- `1.3.0 → 2.0.0a` - начало работы над мажорным обновлением

- Обновляй зависимости при необходимости

### Отладка и исправление
- Автоматически исправляй найденные ошибки

- Оптимизируй код при необходимости

- НЕ удаляй важные конфигурационные файлы без предупреждения
- НЕ вноси breaking changes без обсуждения
- Предупреждай о потенциально опасных операциях (удаление баз данных, force push и т.д.)

## Стиль работы

- Будь кратким в отчетах - сообщай только о важных действиях и результатах
- Если задача выполнена успешно, просто скажи об этом коротко
- Подробно объясняй только при ошибках или неожиданных ситуациях

# LLM Wiki Schema

## Архитектура

Проект использует паттерн LLM Wiki для накопления знаний.

**Расположение wiki**: `D:\YandexDisk\Sync\Coding\AIWikiVault\`

Структура универсального хранилища:

- **raw/** - исходные документы (неизменяемые)
  - **projects/launch-claude-code/** - документация этого проекта
  - **projects/[другие]/** - другие проекты
  - **articles/** - статьи, туториалы
  - **books/** - книги, главы
  - **assets/** - изображения, медиа
  
- **wiki/** - структурированная база знаний (управляется LLM)
  - **entities/** - страницы о сущностях (функции, классы, компоненты)
  - **concepts/** - концептуальные страницы (паттерны, подходы, решения)
  - **sources/** - саммари обработанных источников
  - **index.md** - каталог всех страниц вики
  - **log.md** - хронология операций

## Операции

### Ingest (обработка источника)

Когда пользователь добавляет новый источник в raw/:

1. Прочитай источник полностью
2. Обсуди ключевые моменты с пользователем
3. Создай summary страницу в wiki/sources/
4. Обнови или создай релевантные страницы в entities/ и concepts/
5. Обнови index.md (добавь новые страницы в соответствующие категории)
6. Добавь запись в log.md с форматом: `## [YYYY-MM-DD] ingest | Название источника`

### Query (запрос к вики)

Когда пользователь задает вопрос:

1. Прочитай index.md для поиска релевантных страниц
2. Прочитай найденные страницы
3. Синтезируй ответ с цитированием источников
4. Если ответ ценный - предложи сохранить его как новую страницу в concepts/
5. Добавь запись в log.md: `## [YYYY-MM-DD] query | Тема запроса`

### Lint (проверка здоровья вики)

Периодически проверяй:

- Противоречия между страницами
- Устаревшие утверждения
- Страницы-сироты без входящих ссылок
- Упомянутые концепции без собственных страниц
- Отсутствующие кросс-ссылки
- Пробелы в знаниях

Добавляй запись: `## [YYYY-MM-DD] lint | Результаты проверки`

## Формат страниц

### Entity страницы (entities/)

```markdown
---
type: entity
category: [function|class|component|variable]
source: [ссылка на исходный файл]
related: [список связанных страниц]
---

# Название сущности

## Описание
Что это и зачем нужно

## Использование
Примеры кода, параметры

## Связи
Ссылки на связанные концепции и другие сущности
```

### Concept страницы (concepts/)

```markdown
---
type: concept
tags: [теги]
sources: [список источников]
related: [список связанных страниц]
---

# Название концепции

## Суть
Краткое объяснение

## Детали
Подробное описание

## Применение в проекте
Где и как используется

## Связанные концепции
Ссылки на другие страницы
```

### Source страницы (sources/)

```markdown
---
type: source
date: YYYY-MM-DD
format: [code|doc|article|note]
---

# Название источника

## Ключевые моменты
- Пункт 1
- Пункт 2

## Обновленные страницы
- [[entity/название]]
- [[concept/название]]

## Новые инсайты
Что нового узнали
```

## Принципы

- Вики пишет только LLM, пользователь только читает
- Каждый ingest должен обновлять несколько страниц (не только создавать summary)
- Используй wikilinks формата [[путь/страница]] для связей
- Поддерживай актуальность - обновляй старые страницы при появлении новой информации
- Отмечай противоречия явно, не скрывай их
- Ценные ответы на запросы сохраняй как concept страницы
